Method behind the madness


Before we begin lets address the elephant in the room. "Why, for the love of god, did I choose mongo for this."


The simple answer is, I already know how to use SQL. I have had very limited experience in dealing with non-relational data and this is my last time to do so while I still have highly skilled lecturers available to give me a pole in the correct direction if I get stuck. The reality of th situation is this project would have be far better suited to SQL, however bar a mongoDB pay wall that blocked me linking my PowerBi up to my mongo cloud everything worked out ok anyway. I did have to build a relational database with my mongo info in Power Bi anyway so there's pictures of that available below.

Mongo - What the data collected looks like

It looks like unstructured mongo data with a bunch of Arrays in there for good measure.

Re-structuring Unstructured data

I centered all of the data around the "Projects" table, not unlike the centre of a flower with the remaining tables connected to it like petals, I did have to refactor some code to ensure the user id was added to the Porjects table and set t up so one user can have many Projects. After that it was pretty simple.

PowerBi Dashboard
